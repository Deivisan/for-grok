## Methodology Notes (Training)

### What worked with BrowserOS on this link:
- `browseros_get_page_content` (full page) successfully retrieved the entire conversation text (~11k chars).
- `browseros_evaluate_script` + custom JS was able to split into clean turns with speaker detection (Grok vs Usuário).
- The page is fully loaded and scrollable in one go for short/medium conversations (this one had no lazy loading beyond initial render).
- Speaker distinction was possible by pattern matching (Grok responses often start with "E aí", "Entendi", "Ok, esse é o final", etc.).

### Challenges observed:
- Lots of UI noise (buttons "Copiar", "Report", "14 sources", "Continuar conversa", sidebar elements).
- Need robust cleaning: filter short UI strings, ignore source counts, etc.
- For very long conversations (hours), we will need:
  - Repeated `browseros_scroll` (down) + waits.
  - Re-extraction + merging (detect duplicates).
  - Timeout strategy that is patient but not infinite (e.g., max 30-60 min with progress checks).

- Test with longer real conversations.
- Build a reusable function/script that:
  1. Opens the share link via BrowserOS.

  3. Extracts clean text only.
  4. Splits by speaker (Usuário / Grok) using heuristics + context.
  5. Outputs a single clean .md file.
- Measure time vs conversation length.
- Handle edge cases (very long voice convos, images, sources, etc.).
